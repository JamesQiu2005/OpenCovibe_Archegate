# OpenCovibe 前端配色调整说明（草案）

> 目的：把当前 OpenCovibe 的界面配色，从现在的暖色调（琥珀金主色）调整到贴近公司 VI 的冷色调体系。本文档只给方向、取值和优先级，不含代码实现，具体改动交给 Codex 处理。

---

## 1. 品牌色取值

**来源分两层，官网实际生产环境的取值优先级更高（公司还没有统一 VI 手册，官网是目前唯一"已经上线、有人审核过"的参考）：**

### 1a. Logo 取色（像素采样）

| 名称 | HEX | 用途推测 |
|---|---|---|
| 品牌深蓝 | `#07338C` | logo 主视觉色块，斜切对角线主体 |
| 品牌天蓝 / 青蓝 | `#079DD9` | logo 辅助色块，左右两侧 |
| 品牌金黄 | `#F1D86B` | logo 点缀色块，右上角一小块 |

`logo_dark_mode` 那张采样下来是纯白+透明，无有色像素——是给深色背景用的白色单色版，不携带配色信息。

### 1b. 官网深色模式实际取值（更可信，直接采样自 archebase 官网首页截图）

| 用途 | HEX | 备注 |
|---|---|---|
| 页面背景 | `#10181F` | **关键：带蓝黑调的深色，不是中性灰**（R16 明显低于 B31），说明官方深色场景的背景本身就在往品牌蓝调靠，不是纯灰 |
| 强调蓝（标题高亮文字用色） | `#2B65C5` | 比 logo 天蓝 `#079DD9` 更深、更沉稳，饱和度更低——这是专门为深色 UI 场景调过的一版蓝，跟 logo 蓝不是同一个色，实际做交互色应该以这个为准 |
| 次级表面色（导航激活态底色） | `#1B2835` | 比页面背景亮一档，用于"当前选中/悬浮"这类需要轻微浮起的表面 |
| 主按钮背景 | `#293036` | 中性深灰，非彩色——官网自己的主 CTA 也没有用高饱和蓝或金，走的是克制路线，只用文字色和描边区分层级 |

**结论调整**：官网证据表明公司自己的深色 UI 实践是"背景带蓝黑调 + 强调色用一支专门调低饱和度的深蓝 + 按钮保持中性灰"，跟单纯从 logo 取纯色再套用到界面上是两回事。下面第 3、5 部分按这个新证据调整。

**局限性说明**：logo 三色和官网取值都是截图/导出图采样，不是拿到 CSS 源码或 Figma 文件的精确值。真要定稿建议直接让 Codex 去 view-source 官网页面扒一遍实际 CSS 变量/hex，比截图采样更准；这份文档先按取到的值走。

## 2. 现状审计

当前 OpenCovibe 用的是 shadcn 风格的 CSS 变量系统（`src/app.css` + `tailwind.config.ts`），核心变量：

| 变量 | 浅色模式当前值 | 换算 HEX（约） | 深色模式当前值 |
|---|---|---|---|
| `--primary` | `38 86% 64%` (HSL) | `#f2b854`（琥珀金） | 同色相，同一变量 |
| `--background` | `0 0% 100%` | 纯白 | `0 0% 7.1%`（近黑中性灰） |
| `--ring` | `38 86% 64%` | 同 primary | 同 |
| `--sidebar-primary` | `38 86% 64%` | 同 primary | 同 |
| `--destructive` | `0 64% 57%` | 红色 | 不变（语义色，不建议动） |

**结论**：现在整个产品的强调色系统只有一个主色调（暖色琥珀金），贯穿按钮、链接、进度条、sidebar 高亮、聚焦环。公司 VI 是冷色调为主（深蓝+青蓝），局部点缀暖金——现在的用法正好反过来了：暖金是主角，冷色缺席。这是这次调整要解决的核心方向问题，不是简单地把一个色值换成另一个色值。

## 3. 建议映射方案

不是简单地"把 amber 换成 navy"，而是重新分配三个品牌色各自的角色：

| Token | 建议取值 | 理由 |
|---|---|---|
| `--background` 深色模式 | `#10181F` | **改用官网实测值，推翻上一版"维持中性灰"的判断**——官网证据表明公司自己就是用带蓝黑调的深色背景，不是纯灰，这是目前唯一有生产环境验证的取值 |
| `--primary`（主按钮、链接、聚焦态） | `#2B65C5`（官网强调蓝，而非 logo 天蓝 `#079DD9`） | logo 天蓝饱和度偏高，直接铺到大面积 UI 交互元素上容易显得刺眼；官网这支蓝是专门为深色场景调低饱和度、调深一档的版本，跟 `#10181F` 背景搭配已经过实际验证，直接复用比自己重新调风险低 |
| `--sidebar-primary` / `--ring`（侧边栏高亮、聚焦环） | 同 `#2B65C5`，与 primary 保持一致 | 保持整体强调色统一，不引入第三种交互色造成认知负担 |
| `--secondary` / 次级表面（悬浮、激活态背景） | `#1B2835`（官网导航激活态底色） | 比页面背景亮一档，用于卡片悬浮、当前选中项这类需要轻微浮起但不抢主色风头的场景 |
| logo 深蓝 `#07338C` | 用作更次要的点缀场景（状态 badge、装饰性区域），不要跟 `--primary` 混用 | 这支色跟官网实际交互蓝 `#2B65C5` 不是同一支，混用会让蓝色系统显得不统一，建议只在明确需要"品牌 logo 本色"的地方用，比如关于页面、品牌相关的静态展示 |
| 金黄 `#F1D86B` | **只保留在 logo 本身和极少数点缀位置**（比如"新功能"标记），不做系统级 primary | 官网首页截图里几乎看不到金黄的使用痕迹，进一步佐证这支色在公司实际视觉语言里就是稀缺点缀色，不是系统色 |
| `--destructive`（错误/危险态） | 不动，保持现在的红色 | 语义色不应该因为品牌重塑而改变，用户对"红=危险"的认知不该被打破 |
| 浅色模式 | 官网截图只提供了深色模式的证据,没有浅色模式参考 | 浅色模式配色建议 Codex 按同一套蓝色系（`#2B65C5`）自行推导一版,或者你确认公司是否本来就只做深色模式,不需要维护浅色 |

## 4. 实施优先级

建议分两批，不要一次性全量替换：

**第一批（低风险，先做）**
- `--primary` / `--ring` / `--sidebar-primary` 三个变量统一换成天蓝 `#079DD9`，浅色和深色模式分别调好对比度（深色模式下可能需要提亮一档，比如 `#1FB0E8` 级别，具体由 Codex 跑一遍 WCAG 对比度检查后微调）。
- 检查这几个变量改动后，全站有没有硬编码色值绕过了 CSS 变量系统（比如某个组件直接写死 `#efb043` 或者 `bg-amber-500` 这类 Tailwind 内置色而不是引用 `--primary`）——如果有，这些地方光改变量是不生效的，得单独枚举出来改。

**第二批（需要设计判断，后做）**
- 深蓝、金黄具体在哪些次要场景使用，这个不建议 Codex 自己决定，应该你出一版具体的"使用场景清单"（比如"完成状态用深蓝 badge""设置页某个入口用金黄点缀"）,再让它去实现,避免它自由发挥导致到处滥用金黄。

## 5. 待确认项

- ~~深色模式背景要不要往品牌色靠拢~~——已用官网实测数据解决，答案是"要"，不用再纠结。
- 所有 HEX 仍是截图采样，不是拿到 CSS 源码的精确值。建议第一步就让 Codex 去 view-source 官网首页,直接扒 CSS 变量或计算样式里的准确 hex,比继续用截图取色可靠。
- 官网首页只暴露了首页这一屏的深色配色,没看到浅色模式、也没看到表单/输入框/下拉菜单这类更细粒度组件的配色实践——如果官网有其他页面（比如登录页、后台）用到了这些组件，建议一并截图取色，避免 Codex 在没有参考的场景下自由发挥。
- 官网导航栏用了圆角胶囊按钮（`Home` 按钮的圆角选中态），如果 OpenCovibe 现有的 `--radius: 0.5rem` 跟官网的圆角力度差异明显，也可以顺带让 Codex 对一下，不只是颜色，观感上的"圆润度"也是品牌感的一部分。
